# 🌐 Web RAG

> A lightweight **Retrieval-Augmented Generation (RAG)** pipeline that ingests any web page, embeds its content into a persistent vector store, and answers questions grounded exclusively in that source — fully locally via **Ollama**.

---

##  Features

- **Web Ingestion** — Feed any public URL and automatically load, parse, and chunk its content using LangChain's `WebBaseLoader` and `RecursiveCharacterTextSplitter`.
- **Local Embeddings** — Uses `sentence-transformers` (`all-MiniLM-L6-v2`) to generate embeddings entirely on-device — no OpenAI key required.
- **Persistent Vector Store** — Backed by **ChromaDB** with incremental upsert logic: new chunks are added, changed chunks are updated, and unchanged chunks are skipped.
- **Deduplication via Hashing** — SHA-256 hashes on both source URLs and individual chunks prevent redundant re-indexing across runs.
- **Local LLM Inference** — Answers are generated by **Ollama** (`qwen2.5:3b` by default) with a strict RAG prompt that refuses to hallucinate beyond the indexed context.

---

##  Architecture

```
User Input (URL)
      │
      ▼
 WebBaseLoader          ← fetches raw HTML / text from the URL
      │
      ▼
 RecursiveCharacterTextSplitter   ← chunks text (1000 chars, 200 overlap)
      │
      ▼
 EmbeddingsManager      ← encodes chunks with all-MiniLM-L6-v2
      │
      ▼
 VectorStore (ChromaDB) ← persists embeddings with incremental upsert
      │
      ▼
 RAGRetriever            ← embeds query → cosine similarity search → top-k chunks
      │
      ▼
 Ollama LLM (qwen2.5:3b) ← generates grounded answer from retrieved context
      │
      ▼
      Answer + Sources
```

##  Getting Started

### Prerequisites

| Requirement | Version |
|---|---|
| Python | ≥ 3.13 |
| [Ollama](https://ollama.com) | Latest |
| `qwen2.5:3b` Ollama model | — |
| [uv](https://github.com/astral-sh/uv) *(recommended)* | Latest |

### 1. Clone the repository

```bash
git clone https://github.com/ABHIRAM-AP/Web_RAG.git
cd Web_RAG
```

### 2. Install dependencies

**With `uv` (recommended):**
```bash
uv sync
```

**With `pip`:**
```bash
pip install -r requirements.txt
```

### 3. Pull the Ollama model

```bash
ollama pull qwen2.5:3b
```

### 4. Configure environment variables

Create a `.env` file in the project root:

```env
# Required only if using the web_search tool (tools.py)
LANGSEARCH_API_KEY=your_langsearch_api_key_here
```

### 5. Run

```bash
python main.py
```

You will be prompted to enter a URL:

```
Enter a url to feed: https://cloud.google.com/discover/what-is-retrieval-augmented-generation
```

The pipeline will:
1. Fetch and chunk the page content
2. Embed and store chunks in ChromaDB (incremental — safe to re-run)
3. Answer the hardcoded question against the indexed content

##  Configuration

| Parameter | Location | Default | Description |
|---|---|---|---|
| Embedding model | `src/loader.py` | `all-MiniLM-L6-v2` | Sentence-transformer model name |
| LLM model | `src/llm.py` | `qwen2.5:3b` | Ollama model tag |
| LLM temperature | `src/llm.py` | `0.2` | Controls response randomness |
| Chunk size | `main.py` | `1000` | Characters per text chunk |
| Chunk overlap | `main.py` | `200` | Overlap between adjacent chunks |
| Top-K retrieval | `src/retriever.py` | `5` | Number of chunks retrieved per query |
| Score threshold | `src/retriever.py` | `0.0` | Minimum cosine similarity to include a chunk |
| Vector store path | `src/vector_store.py` | `data/vector_store` | ChromaDB persistence directory |
| Collection name | `src/vector_store.py` | `url_info` | ChromaDB collection name |

##  How It Works

### Incremental Indexing

Each source URL is hashed (SHA-256) to create a `file_hash`. Every chunk within that source gets its own `chunk_hash`. On subsequent runs with the same URL:

- **Unchanged chunks** → skipped (no re-embedding)
- **Modified chunks** → updated in place
- **New chunks** → added to the collection

This makes re-indexing fast and idempotent.

### Retrieval & Answer Generation

1. The user's query is embedded with the same `all-MiniLM-L6-v2` model.
2. ChromaDB performs a cosine-similarity search and returns the top-K chunks.
3. Retrieved chunks are assembled into a context string with their source metadata.
4. The LLM is prompted with a strict instruction: **answer only from the provided context**.
5. If no relevant chunks are found, the system returns a fallback message rather than hallucinating.
